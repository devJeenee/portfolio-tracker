# Git 개발 스토리 분석

Git 히스토리에서 트러블슈팅, 리팩토링, 아키텍처 변경 등의 개발 스토리를 찾습니다.

## 실행할 작업

`analyze_git` MCP 도구를 사용하세요:

- `path`: $ARGUMENTS (비어있으면 현재 디렉토리)

## 출력 형식

발견된 개발 스토리를 한국어로 정리해주세요:
- 각 스토리의 배경(왜 했는지) + 과정(어떻게 했는지) + 결과(무엇이 좋아졌는지)
- 포트폴리오/면접에서 활용할 수 있는 서술 포인트

## 에러 대응

- **Git 저장소가 아닌 경우**: "이 디렉토리는 Git 저장소가 아닙니다. Git 히스토리가 있어야 개발 스토리를 분석할 수 있습니다. `git init`으로 저장소를 초기화하거나, Git 저장소 경로를 지정해주세요." 라고 안내
- **커밋이 10개 미만**: "커밋이 너무 적어(N개) 의미 있는 개발 스토리를 추출하기 어렵습니다. 더 많은 커밋이 쌓인 후 다시 시도해주세요. 현재 커밋으로 가능한 분석만 진행합니다." 라고 안내
- **커밋 메시지가 대부분 의미없는 경우** (예: 'update', 'fix', 'wip' 등이 80% 이상): "커밋 메시지가 대부분 단순하여 스토리 추출이 어렵습니다. 향후 커밋 시 'feat: 사용자 인증 기능 추가', 'fix: 결제 금액 계산 오류 수정' 처럼 의미 있는 메시지를 작성하면 더 풍부한 분석이 가능합니다. 현재 커밋으로 가능한 분석만 진행합니다." 라고 안내
- **path에 pom.xml / build.gradle 없음**: "이 경로에서 Spring Boot 프로젝트를 찾을 수 없습니다. Spring Boot 프로젝트의 루트 디렉토리를 지정해주세요." 라고 안내
- **분석 결과가 비어있는 경우**: "개발 스토리를 추출할 수 없습니다. 프로젝트 경로와 Git 히스토리를 확인해주세요." 라고 안내

## 후속 제안

- "발견된 스토리를 면접 답변으로 다듬고 싶다면, 스토리 번호를 알려주세요. STAR 형식의 면접 스크립트를 작성해드립니다."
- "코드 레벨의 상세 분석이 필요하다면 `/analyze`로 코드 품질을 확인해보세요."
- "Git 히스토리 기반 경험을 포트폴리오용으로 정리하고 싶다면 `/experiences`를 사용해보세요."
- "전체 포트폴리오 가치가 궁금하다면 `/portfolio`로 종합 분석을 받아보세요."
